AI_Writer_TISTORY는 티스토리용 글쓰기 자동화가 왜 단순한 생성기보다 운영형 백엔드에 가까운지 보여 주는 저장소입니다. 초안 생성, 프롬프트 로그, SQLite 기록, 발행 준비 상태를 따로 분리하지 않고 하나의 루프로 묶어 두어 개인 운영자가 반복 작업을 줄이면서도 기록을 놓치지 않게 설계했습니다.

글쓰기 자동화보다 운영 자동화에 더 가깝습니다

생성 결과만 남는 구조가 아니라, 운영자가 다음 액션을 이어갈 수 있는 기록 중심 백엔드로 설명되는 페이지입니다.

01

티스토리 자동화 서버를 독립 레포로 분리

글쓰기 자동화 로직을 다른 실험용 코드와 섞지 않고, 발행 루프 중심 저장소로 또렷하게 관리합니다.

02

Prompt log와 SQLite로 운영형 백엔드 구성

무엇을 생성했는지뿐 아니라 어떤 운영 맥락에서 만들었는지를 같이 저장해 회고가 가능하게 했습니다.

03

Notion 운영 로그와 발행 파이프라인 연결

콘텐츠 초안, 운영 메모, 발행 준비 체크를 같은 흐름으로 읽을 수 있게 정리해 검색 유입에도 설명력이 높습니다.

Chapters

프로젝트를 이해하는 핵심 3단계

저장소 안의 Notion 문서 제목을 그대로 살려, 이 프로젝트가 어떤 순서로 설계되었는지 블로그 글처럼 읽히게 구성했습니다.

1. 티스토리 자동화 서버를 별도 레포로 분리한 이유

글 생성 로직이 아니라 운영 루프 자체를 저장소의 중심으로 두기 위한 출발점입니다.

2. Prompt log와 SQLite를 붙여 백엔드를 운영형으로 만든 과정

반복 실행과 기록 축적을 고려해 단순 API 서버보다 운영 콘솔에 가까운 구조로 발전시켰습니다.

3. 티스토리 발행 파이프라인과 Notion 운영 로그를 묶은 결과

초안 생성과 발행 준비 사이에 남는 운영 공백을 문서와 로그로 메우는 방식입니다.

Operating notes

실행 화면 대신 운영 포인트를 먼저 보여 주는 페이지

아직 런타임 캡처보다 구조 설명이 더 중요한 저장소라서, 사용자가 가장 빨리 이해해야 할 운영 포인트를 본문 안에 카드 형태로 녹였습니다.

STEP 1 환경 변수와 API 키를 먼저 맞춥니다

`.env.example`을 기준으로 OpenAI와 운영용 환경 변수를 정리한 뒤 개발 서버를 띄우는 흐름을 전제로 합니다.

STEP 2 초안 생성과 운영 로그를 함께 확인합니다

생성 결과만 확인하는 것이 아니라 prompt log와 SQLite 기록을 같이 읽는 것이 핵심입니다.

Quick start

`npm install`, `npm run dev`, 필요 시 `npm run build`로 백엔드 상태를 점검합니다.

Repository structure

`src/`, `scripts/`, `dashboard/`, `docs/`를 나눠 실행 로직과 운영 문서를 분리했습니다.

FAQ

자주 묻는 질문

AI_Writer_TISTORY는 무엇을 자동화하나요?

티스토리용 AI 글쓰기 초안과 운영 로그 축적, 발행 준비 흐름을 자동화합니다.
